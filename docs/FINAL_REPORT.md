## 摘要

「詩境」是一個結合 Gemini API 與本機唐詩資料庫的互動式古典詩詞系統。使用者可輸入作者、篇名、詩句片段或意境描述，系統會先以本機 `poems.json` 查詢作品；若 API Key 可用，則使用 Gemini 進行延伸分析與問答。專案的核心目標不是單純查詩，而是讓詩作以「詩魂現身」的形式被閱讀，將原詩、白話釋義、情緒、背景、朗讀、收藏與雙詩人對話整合在同一個網頁體驗中。

本專案從初版 AI 查詩頁面，逐步改版為可部署、可展示、可離線備援且資料輸出較穩定的純前端網站。過程中主要解決 Gemini 格式不穩、API 配額限制、手機版排版、詩庫原始資料混雜、校注與譯文標籤誤顯示、GitHub Pages 快取等問題。最終版 v1.5.0 整合本機詩庫、API 狀態、隨機召喚、朗讀、收藏、詩魂問答、雙魂對話、校注分離、輸出清理與完整版本紀錄，作為期末展示版本。

## 一、我做了什麼

我完成了一個名為「詩境」的古典詩詞互動網站。它可以讓使用者用自然語言或關鍵字召喚詩作，並以較有氛圍的介面呈現原詩、詩魂自述、情緒分析、背景資料與延伸互動。

主要功能包括：

- **通靈問詩**：輸入作者、篇名、詩句片段或意境描述，系統回傳詩作、白話釋義、情緒與背景。
- **本機詩庫備援**：使用 poems.json 儲存 315 首詩，無 API Key 或 API 失效時仍能展示。
- **Gemini API 生成**：當 API 可用時，可做延伸分析、意境查詢與詩魂問答。
- **隨機召喚**：不輸入文字也能抽出一首詩，適合 demo 與探索。
- **詩魂問答**：針對目前詩作追問，讓詩人以第一人稱回應。
- **雙魂對話**：選擇兩位詩人與主題，生成跨詩人對話。
- **朗讀**：使用瀏覽器內建 Web Speech API，不需外部 TTS 服務。
- **收藏與複製**：收藏存 localStorage，複製可輸出詩名、作者與原文。
- **校注分離**：將「唯 通：惟」「明年 一作：年年」等資料從原詩中抽出。

## 二、為什麼做這個專案

古典詩詞在一般學習情境中，常被拆成作者、注釋、翻譯與考題。這種方式雖然有效率，但容易讓詩變成資料，而不是一段情緒與歷史經驗。另一方面，單純使用 AI 生成詩詞解釋又會有幻覺、格式不穩、資料來源不透明的問題。

因此我把專案目標設定為三點：

1. **降低閱讀門檻**：讓使用者不用先查完整詩名，也能用片段或意境找到作品。
2. **保留詩的氛圍**：用直排、淡色背景、詩魂語氣與朗讀功能，讓閱讀不只是資料查詢。
3. **提高系統可信度**：用本機詩庫作為穩定資料來源，API 則用於延伸分析與互動。

## 三、系統架構

本專案採純前端架構，適合 GitHub Pages 部署，不需要後端伺服器。

| 檔案 | 功能 |
|---|---|
| `index.html` | 網頁結構與主要區塊 |
| `style.css` | 視覺設計、直排詩文、手機版排版 |
| `app.js` | 查詢、API、本機詩庫、快取、收藏、朗讀、互動邏輯 |
| `poems.json` | 本機唐詩資料庫，共 315 首 |
| `README.md` | 專案介紹與使用方式 |
| `USER_GUIDE.md` | 使用者指南 |
| `RELEASE_NOTES.md` / `CHANGELOG.md` | 版本更新紀錄 |
| `docs/` | 專題報告、設計理念與更新摘要 |

查詢流程如下：

```text
使用者輸入
→ 本機 poems.json 查詢
→ 若命中：以本機資料輸出
→ 若未命中且 API 可用：呼叫 Gemini
→ 若 API 不可用：提供本機相近查詢、隨機召喚或重試
```

## 五、遇到的問題與解法

| 問題 | 影響 | 解法 |
|---|---|---|
| AI 生成不穩定 | 僅依賴 Gemini 可能遇到格式漂移或查不到本詩。 | 加入本機 poems.json，並在 prompt 與前端解析中限制欄位輸出。 |
| API 配額與測試過度 | 自動測試 API Key 若沒有節制，可能造成 429。 | 加入同 key 去重、最小測試間隔與 429 冷卻，狀態列明確顯示。 |
| 詩庫原始資料混雜 | 原詩 content 混入校注、題序、譯文標籤或英文翻譯。 | 新增資料清洗與前端輸出清理，分離校注、長序文與中文譯文。 |
| 瀏覽器快取 | GitHub Pages 更新後，使用者仍可能讀到舊 app.js 或 poems.json。 | 加入資源版本參數與 poems.json 時間戳，降低舊快取風險。 |
| 手機展示不穩 | 手機導覽、白框高度與按鈕位置容易擠壓。 | 重新調整響應式樣式與導覽邏輯，短詩不再撐出過大空白。 |

## 六、最重要的改善：從能跑到能發布

初版其實已經能做到「輸入 → AI 回答」，但離可展示還有距離。真正花最多時間的是穩定性與資料清理。例如《黃鶴樓送孟浩然之廣陵》原始資料中曾出現：

```text
孤帆遠影碧空盡，唯見長江天際流。 (唯 通：惟)
```

這不是原詩，而是校勘註記。若直接放進直排原詩區，畫面會變形，也會讓使用者誤以為括號內容是正文。因此 v1.4.0 後改成：

```text
原詩：孤帆遠影碧空盡，唯見長江天際流。
校注：唯，通「惟」。
```

後續又發現「譯文及注釋」「英譯」「註解」等原始資料標籤會跑進「詩魂自述」或「魂歸何處」，因此 v1.4.1 與 v1.5.0 進一步清理輸出格式，讓前端只顯示整理後的中文內容。這是本專案從 prototype 走向 release-ready 的關鍵。

## 七、Demo 展示流程

建議展示 5 分鐘，流程如下：

1. **開場說明**：介紹「詩境」不是一般查詩工具，而是詩詞互動空間。
2. **本機查詢**：輸入 `黃鶴樓送孟浩然之廣陵`，展示原詩、詩魂自述與校注分離。
3. **隨機召喚**：按「隨機一首」，展示沒有 API Key 也能使用。
4. **朗讀與收藏**：播放原詩朗讀，並收藏詩作。
5. **API 狀態**：說明 API Key 只存在 sessionStorage，配額不足仍可使用本機詩庫。
6. **雙魂對話**：選兩位詩人，輸入主題，展示 AI 角色互動。
7. **版本反思**：說明從 v1.0.0 到 v1.5.0 的改版，重點是穩定性、資料清理與使用者體驗。

## 八、反思：學到了什麼

這次專案最大的收穫不是「把 API 接上去」，而是理解到一個 AI 專案要能真的展示，需要處理很多 API 之外的細節。

第一，AI 生成內容必須被約束。只靠 prompt 不夠，前端仍要檢查欄位、清理輸出、處理失敗狀態。第二，資料品質會直接影響使用者體驗。`poems.json` 裡的校注、英譯、註解標籤若未處理，介面就會出現不合理輸出。第三，部署後的快取也是真實問題。即使本機修改成功，GitHub Pages 或瀏覽器仍可能讀到舊檔案，所以版本參數與 no-cache 設計很重要。

我也學到，作品的特色不一定是功能數量最多，而是能不能說清楚「為什麼這樣設計」。本專案的特色是把古典詩詞、生成式 AI、本機資料備援、詩魂角色化與安靜的閱讀介面整合在一起，讓使用者能以比較自然的方式接近詩。

## 九、目前限制與未來改進

目前本機詩庫主要以唐詩為主，宋詞與更廣泛的古典文本仍不足。未來可加入宋詞、詩人年表、主題分類、學習紀錄、課堂展示模式與更完整的資料來源標記。AI 問答也可以加入引用依據，讓回答更可驗證。

若繼續發展，本專案可以從「詩詞互動展示」擴充為「古典文學學習平台」，包含詩人頁面、意象地圖、情緒分類、背誦模式與班級作業紀錄。

## 十、結論

「詩境」最終完成的不只是查詢功能，而是一個具有明確設計語氣、可離線備援、可 API 延伸、可展示、可發布的生成式 AI 網頁作品。它的核心價值在於：用 AI 降低詩詞理解門檻，用本機資料提高穩定性，再用介面與角色設定保留古典詩詞的閱讀氛圍。
