# 詩境｜生成式 AI 古典詩詞互動系統 專案報告

## 一、專案主題

「詩境」是一個以古典詩詞為核心的互動式網頁系統。使用者可以輸入作者、篇名、詩句片段或意境描述，系統會先從本機 `poems.json` 詩庫查找作品；若本機沒有命中，則可使用 Gemini API 生成詩詞解讀內容。作品目標不是單純查詩，而是讓古典詩詞以「詩魂現身」的方式被重新閱讀。

## 二、問題意識

古典詩詞常見的學習方式偏向背誦、註釋與考題整理，容易讓讀者只看見字面意思，卻忽略詩人的情緒、時代背景與作品的美感。本專案想解決三個問題：

1. 查詩工具通常缺少互動性，使用者不容易進一步追問。
2. AI 生成內容可能不穩定，因此需要本機詩庫作為可控資料來源。
3. 傳統詩詞介面多半偏資料表或文章頁，不容易呈現詩詞本身的氣氛。

## 三、版本演進與改版整合

本專案從初版的 AI 查詩頁面，逐步改成可部署、可離線備援、可展示且較穩定的純靜態網站。以下整合各階段更新。

### 1. 初版：通靈問詩與詩魂互動

初版完成使用者輸入作者、篇名、詩句片段或意境描述後，由 Gemini API 回傳詩名、作者朝代、原文、白話釋義、情緒分析、創作背景與詩魂第一人稱回應。此階段建立了「查詩不是查資料，而是召喚詩魂」的核心體驗。

### 2. 結構改版：拆分檔案與本機詩庫

後續將原本集中於 `index.html` 的樣式、邏輯與詩詞資料拆分為 `style.css`、`app.js` 與 `poems.json`。本機詩庫改為 lazy-load，只有需要時才載入，讓網站更適合 GitHub Pages 部署，也方便後續維護與共編。

### 3. 體驗改版：隨機召喚、朗讀與收藏整理

新增「🎲 隨機一首」，可直接從本機詩庫抽取作品，不消耗 Gemini API。朗讀功能採用瀏覽器內建 Web Speech API，不需要 ElevenLabs、不需要額外語音 API Key，也不依賴後端伺服器。收藏入口也重新整理，避免手機版出現重複詩藏按鈕。

### 4. 穩定性改版：API 狀態與錯誤流程

API Key 改為只存 `sessionStorage`，關閉分頁後清除。貼上 key 後採延遲輕量測試，並加入同 key 去重、最小測試間隔與 429 冷卻，避免過度請求。當 API 不可用時，系統會提供重試、切換 API Key、本機相近作品、隨機召喚或重新輸入等下一步。

### 5. 手機版與版面修正

修正手機版導覽列、操作按鈕與詩文白框高度。短詩不再硬撐過高，隨機召喚也不再覆蓋 Gemini API 連線狀態。這些調整讓手機展示時更穩定。

### 6. 資料輸出修正：校注／異文分離

部分詩詞資料的原詩欄位混入「一作」、「通」、「又作」等校勘註記，例如「唯 通：惟」。新版新增自動拆分機制，原詩區只顯示真正詩句，異文與校勘資訊集中顯示在「校注／異文」區，避免註記混入直排詩文造成版面錯誤。

### 7. 快取與發布修正

修正 `poems.json` 舊快取問題。讀取本機詩庫時會自動加入版本與時間戳，使用者不需要手動在網址後加 `?v=`。`index.html` 也對 `app.js` 與 `style.css` 加入版本參數，降低 GitHub Pages 或瀏覽器載入舊版資源的機率。

## 四、系統功能

### 1. 通靈問詩

使用者輸入詩名、作者、詩句或意境後，系統會顯示詩名、作者朝代、原文、白話釋義、情緒分析、創作背景與來源標籤。來源會明確標示為「Gemini」或「本機詩庫」。

### 2. 本機詩庫備援

本機資料使用 `poems.json`，目前主要收錄唐詩。當沒有 API key、API 配額受限或 Gemini 回應不穩定時，系統仍可查詢本機詩庫中的作品。

### 3. 隨機召喚

使用者可以不輸入任何文字，直接按「🎲 隨機一首」探索本機詩庫。這降低了初次使用門檻，也方便展示時快速產生案例。

### 4. 免費朗讀

使用者可以朗讀詩文原文與詩魂自語。此功能不依賴外部 TTS 服務，適合 GitHub Pages 純靜態部署。

### 5. 詩魂問答

使用者可針對目前顯示的作品繼續提問。若 Gemini API 可用，會以詩人第一人稱回應；若 API 不可用，則改用本機詩作資料產生簡短回覆。

### 6. 雙魂對話

使用者可選擇兩位詩人與一個主題，讓系統生成兩位詩人之間的對話。此功能展示 AI 角色設定與古典文學人物語氣模擬。

### 7. 詩藏收藏

使用者可收藏查詢過的詩作，資料存在瀏覽器 `localStorage`，方便在同一台裝置上再次閱讀。

## 五、技術架構

| 模組 | 技術 | 用途 |
|---|---|---|
| 前端頁面 | HTML | 頁面結構與語意標記 |
| 視覺介面 | CSS | 古典紙本風格、直排詩文、響應式設計 |
| 互動邏輯 | JavaScript | 搜尋、隨機召喚、收藏、問答、朗讀 |

| 部署 | GitHub Pages | 純靜態網站發布 |

## 六、實作特色

1. **本機優先**：先查 `poems.json`，減少 API 消耗。
2. **來源透明**：每次顯示結果都標示來源，避免使用者混淆 AI 與本機資料。
3. **低門檻體驗**：不輸入 API key 也能查唐詩、隨機召喚與朗讀。
4. **錯誤可操作**：API 失敗時提供重試、切換 API Key、推薦本機類似文章等下一步。
5. **受控 API Key 測試**：Gemini key 貼上後會延遲輕量測試一次，並用 debounce、同 key 去重與 429 冷卻避免 Too many requests。
6. **資安考量**：不把 API Key 寫死在程式碼，也不保存到長期 localStorage。
7. **古典 UI**：使用直排詩文、紙色背景、朱紅收藏狀態與古銅色線條建立詩詞閱讀氛圍。

## 七、成果展示方式

展示時可依序操作：

1. 開啟 GitHub Pages 或本機 localhost 網址。
2. 不輸入 API Key，查詢「靜夜思」或「石壕吏」，展示本機詩庫功能。
3. 按「🎲 隨機一首」，展示不消耗 API 的本機探索功能。
4. 按詩文旁的「朗讀」，展示免費瀏覽器朗讀。
5. 收藏一首詩，再切到「詩藏」確認收藏。
6. 若需要 AI 功能，再貼上 Gemini API Key；系統會先自動輕量測試 key，通過後再查詢抽象意境或進行詩魂問答。
7. 開啟「雙魂對話」，讓李白與杜甫談「月亮」或「離別」。

## 八、限制與改進方向

目前本機詩庫主要以唐詩為主，宋詞與其他朝代作品不一定能在本機命中。未來可以擴充詩庫，加入朝代篩選、全文搜尋、詩人頁面、學習進度紀錄與更多可視化分析。

瀏覽器朗讀的音色取決於使用者裝置，品質不如專業 TTS 服務。未來若需要更自然的真人語音，可再加入 ElevenLabs 或其他 TTS 作為進階選項。

API 方面，前端直連 Gemini 適合課程展示與個人使用；若要公開給大量使用者，應改成後端代理，避免 API Key 暴露與配額被濫用。

## 九、學習反思

本專案讓我練習如何把生成式 AI 與傳統資料庫結合，而不是完全依賴 AI 回答。後續改版進一步讓系統在沒有 API Key 的情況下仍能完成查詩、隨機探索、收藏與朗讀，並透過 API 狀態管理、手機版排版修正、校注分離與快取處理提升穩定性。過程中也理解到 UI/UX、錯誤處理、API Key 安全、資料清洗與部署結構都會影響作品是否能真正被使用。


## v1.4.1｜本機詩庫輸出清理

本版針對本機詩庫的輸出格式進行修正。過去部分資料欄位同時包含「譯文及注釋」「英譯」等原始來源標籤，可能在前端的「詩魂自述」或「魂歸何處」中被直接顯示，造成閱讀體驗不佳。本次修正新增中文優先與英譯略過邏輯，讓白話釋義只顯示中文譯文，創作背景則優先使用中文背景資料，若無背景才回退到中文賞析。


## v1.5.0 輸出清理補強

本次補強針對原始資料中「註解」與 inline 注釋格式進行處理，避免譯文欄位的註解文字被誤放入「詩魂自述」。同時將部分長篇作品的題序從 `content` 移至 `others`，使原詩直排區只呈現詩句本體。


## 十、v1.5.0 發布檢查與文件整合

本次發布版除了修正輸出邏輯，也同步更新專案說明文件。根目錄的 `README.md` 說明目前版本、功能與使用方式；`USER_GUIDE.md` 說明使用者操作流程與測試建議；`RELEASE_NOTES.md` 與 `CHANGELOG.md` 保留完整版本演進；`docs/UPDATE_SUMMARY.md` 彙整本次修改檔案、檢查結果與發布前測試清單。

本版檢查項目包含：`poems.json` 可正常解析、詩作數量維持 315 首、`app.js` 語法檢查通過、原詩 `content` 中未發現校注括號、譯文標籤、注釋標籤、英譯或長篇題序殘留。
