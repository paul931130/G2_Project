# DEMO_VIDEO_SCRIPT｜詩境 Demo 影片與現場展示腳本

> 建議影片長度：5～6 分鐘  
> 展示版本：v1.4.1  
> 展示重點：功能完整性、資料清理、AI 應用、版本演進與反思

## 0:00–0:30｜開場與專案介紹

畫面：網站首頁與標題。

旁白：

> 本專題名稱是「詩境」，是一個結合古典詩詞、本機資料庫與生成式 AI 的互動網站。使用者可以輸入詩名、作者、詩句片段或意境描述，系統會呈現原詩、白話釋義、情緒意象與背景，也能在 API 可用時進行延伸互動。

> 專案的核心目標不是單純查資料，而是把古典詩詞整理成一個可閱讀、可互動、可展示的 AI 應用。

## 0:30–1:20｜本機詩庫查詢

操作：搜尋 `黃鶴樓送孟浩然之廣陵`。

旁白：

> 系統內建本機唐詩資料庫，目前收錄 315 首作品。即使沒有 API Key，也可以完成基本查詢與展示。

> 這裡搜尋《黃鶴樓送孟浩然之廣陵》，可以看到詩名、作者與原詩正文。系統會優先使用本機詩庫，因此展示時不會完全依賴 API。

## 1:20–2:00｜校注與正文分離

操作：展示《黃鶴樓送孟浩然之廣陵》或《送別》。

旁白：

> 開發過程中遇到一個資料問題：有些詩句後面會混入校勘註記，例如「唯 通：惟」或「明年 一作：年年」。如果直接放在原詩直排區，版面會被破壞。

> 因此系統加入校注分離機制。原詩區只保留正文，異文註記會移到校注區，讓閱讀更乾淨。

## 2:00–2:50｜白話釋義輸出清理

操作：搜尋 `怨情`。

旁白：

> 第二個問題是譯文資料裡常有「譯文及注釋」「注釋」「英譯」等原始標籤。這些內容如果直接輸出，就會讓畫面像未整理的爬蟲資料。

> 在這個版本中，系統會自動清理這些標籤，只保留中文白話內容。像《怨情》這首詩，就不會再把英文翻譯或資料標題顯示在「詩魂自述」區。

## 2:50–3:35｜長詩閱讀與白框伸縮

操作：搜尋 `琵琶行`。

旁白：

> 長篇作品是另一個排版挑戰。如果固定白框高度，直排欄位會被擠得很密，閱讀體驗不好。

> 目前版本保留古典直排形式，但原詩白框會依最長詩句自動垂直伸縮。這樣在展示《琵琶行》這類長詩時，版面不會被壓縮成難以閱讀的狀態。

## 3:35–4:15｜互動功能展示

操作：依序點擊隨機召喚、朗讀、收藏、複製。

旁白：

> 除了查詢，系統也加入隨機召喚、朗讀、收藏與複製功能。隨機召喚可以快速從本機詩庫抽取作品；朗讀使用瀏覽器內建語音，不需要額外付費服務；收藏則使用 localStorage 儲存在使用者自己的瀏覽器。

## 4:15–4:55｜Gemini API 與備援設計

操作：展示 API Key 區與狀態列；若可用，展示詩魂問答或雙魂對話。

旁白：

> 本專案支援 Gemini API，可以進行延伸解讀、詩魂問答與雙魂對話。但 API Key 只會存在目前分頁的 sessionStorage，不會寫進專案檔案，也不會上傳到 GitHub。

> 如果 API 配額受限，系統會顯示狀態並回到本機詩庫模式。這樣即使 API 暫時不可用，展示也不會中斷。

## 4:55–5:40｜版本演進與反思

操作：開啟 `CHANGELOG.md` 或 `docs/FINAL_REPORT.md`。

旁白：

> 本專案從初版一路更新到 v1.4.1。前期重點是完成基本查詩與 AI 互動；中期加入本機詩庫、隨機召喚、朗讀、收藏與雙魂對話；後期主要處理穩定性與資料品質，包括 API 配額、快取、手機版排版、校注分離、譯文標籤清理與長詩白框伸縮。

> 最大的反思是，生成式 AI 專案不是接上 API 就完成。真正影響作品品質的是資料整理、錯誤處理、使用者體驗與版本管理。

## 5:40–6:00｜結尾

畫面：回到網站首頁或展示一首詩。

旁白：

> 「詩境」的核心特色，是讓古典詩詞不只是被查詢，而是以互動、朗讀、分析與詩魂對話的方式重新被閱讀。這是本專題想呈現的生成式 AI 應用價值。

## 現場展示備案

若 Gemini API 無法使用，展示流程可改為：

1. 本機詩庫查詢。
2. 隨機召喚。
3. 校注分離。
4. 白話釋義清理。
5. 長詩白框伸縮。
6. 朗讀、收藏與複製。
7. 說明 API 備援設計。

此流程不依賴外部 API，仍可完整展示專案核心成果。
